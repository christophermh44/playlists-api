Audio files definition

An audio file is defined by the following fields:

id
The unique ID of the audio file
category
The category of the audio file (similar to a folder where the file is)
alias
A unique alias to easily get the file when querying it
title
The title of the audio file
artists
The list of artists of the audio file
duration
The duration of the file (between mix points)
tags
The list of tags of the audio files

Querying the database

You may use the file querrier to get a file, specifing the filters inside the backticks (`...`).

The following examples shows how to get the file by its id or its alias: 

jingle = file`id: 5`
toth = file`alias: toth`

The file notation will always give only one file. If you want to get a collection of file sharing the same criteria, you may use the collection notation as following: 

jingles = collection`category: jingles`

You may combine multiple criteria to refine your search. Note that all criteria mentionned must match to allow the file to be set in the collection: 

songs_from_foobar = collection`
    category: musics
    artists: foobar
`

Defining repeat rules

Repeat rules are rules that tell the algorithm how to pick files when it creates the playlist. It avoids a file to be choosen twice in a row for example.

To define repeat rules, you may use the picker notation as following: 

songs_picker = picker`
    delay_before_same_artist: 45 minutes
    delay_before_same_file: 90 minutes
    on_no_candidates: pick_oldest
`

The criteria inside backticks may be the following:

delay_before_same_artist
The time during when an artist cannot be picked twice
delay_before_same_file
The time during when a file cannot be picked twice
on_no_candidate
(NOT YET IMPLEMENTED)

Defining loops

A loop is a set of files, categories or pickers defining what kind of audio files the algorithm can choose successively.

You may define a loop using the following notation (using braces ([...])): 

music_rotation = [
    jingles_picker(jingles),
    songs_picker(songs)
]

The example above defines a loop that will play a jingle following by a song, repeatedly.

Interruptions

Interruptions are everything that will interrupt the algorithm in its generation. For example, an interruption may be used to place a "Top of the Hour" at each hour change.

The easiest way to place a Top of the Hour in a playlist is to consider that it has to be prepared from HH:58 (considering a song may last about 2~4 minutes, so the Top of the Hour will be played between HH:58 and HH:02).

Here is an example of code that will play the Top of the Hour when needed: 

prepare_toth = ({ currentRotation, queue, next }) => {
    if (next.belongsTo`category: jingles`)
        replace(queue, -1, toth)
    else
        playNext(queue, toth)
    currentRotation.index = 1
}

when`
    seconds: 0
    minutes: 58`
(prepare_toth)

First, we create a function that will be executed when the Top of the Hour is needed. This function checks if the next item in the playlist is a jingle and replaces it by the Top of the Hour (we do not want to play a jingle followed by the Top of the Hour) or else, it just plays the Top of the Hour next. Finally, it sets the current rotation index to 1 to ensure the next element picked will be a song (following what we defined in the music rotation, considering that the loop indexes are 0-based).

The when notation defines crieteria like an alarm-clock. The criteria may be the following:

seconds
minutes
hours
You may define these criteria to define when the interruption will be triggered. Consider that if you do not write one of them, the algorithm will consider that all seconds, minutes or hours are valid. That's how, for the example of the Top of the Hour above, hours is not defined, so it is triggered each hour.
You may define a range using a hyphen: 10-20 (all between 10 to 20), or specific values using commas: 10,20,30 (only at 10, 20, and 30).
You may also tells something like "every 5 (hours or minutes or seconds)" using this notation: */5.
days_of_month
Like seconds, minutes, and hours, defines the days of the month (from 1 to 31).
months
Defines the months (from 1 to 12, or using aliases like JAN-DEC)
days_of_week
Defines the days of the week (from 0 to 7, 0 and 7 are both for Sunday, or using aliases like SUN-SAT)

Then, let's play!

To finish, you may indicate the algorithm what will be his main loop it will work with: 

run(music_rotation)

About

This demo shows how musical playlists (for radio stations, for example) can be generated programmatically.

Using a fictional database, the application allows you to define playlist generation rules in the central editor. Clicking the generation button creates a fictional playlist based on the data from that database.

In this demonstration, I wanted to focus heavily on the user experience. The scripting language is nothing more, nothing less than JavaScript. However, the syntax choices I made for this playlist generation language aim to make playlist programming accessible even to people who are not familiar with software development.
